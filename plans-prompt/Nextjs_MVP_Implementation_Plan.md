# Next.js 기반 MVP 구현 계획 (Implementation Plan)

본 문서는 `SRS_v0_1_Opus.md`에 명시된 비즈니스 요구사항을 최적화된 모던 웹 스택(Next.js App Router, Supabase, Vercel AI SDK, Google Gemini)으로 단일화하여 구현하기 위한 상세 작업 계획 및 사용자 경험 검토 결과를 정의합니다.

---

## 1. 핵심 사용자 경험 (가치 전달) 검토 결과

기술 스택 전면 개편(단일 풀스택, Vercel Serverless, 증권사 REST API 한정)이 MVP의 핵심 가치(감정 제어, 자동 복기)에 미치는 영향을 검토한 결과입니다.

### 1) 실시간 팩트폭행 알람 (F1)
* **변경점:** 상시 구동되는 데몬 서버의 부재로 인해, 실시간 감시 주체를 'Vercel Cron(1분 주기)'과 '클라이언트(브라우저) 활성 시 폴링'으로 이원화하고 알림 전송을 'Supabase Realtime(웹소켓)'으로 대체.
* **가치 훼손 여부: 없음 (또는 미미함)**
  * 사용자가 모니터(웹앱)를 보고 매매하는 동안에는 클라이언트 주도 감시 + Supabase Realtime으로 기존의 '3초 이내 팝업' UX를 동일하게 제공할 수 있습니다.
  * 백그라운드 상태에서는 1분 주기로 감시되나, 주식/선물 매매 호흡을 고려할 때 1분의 알람 지연이 '뇌동매매 차단'이라는 근본 가치를 크게 훼손하지 않습니다.

### 2) 멀티 증권사 통합 연동 (F2)
* **변경점:** Windows OS(OCX)가 필수인 키움증권을 배제하고, 순수 REST API를 지원하는 **한국투자증권, 토스증권, LS증권** 등으로 타겟 전환.
* **가치 훼손 여부: 일부 타겟 유입 지연 가능성 있으나 수용 가능한 리스크**
  * 키움증권 사용자의 초기 진입이 제한된다는 단점이 있으나, "자신의 매매를 통제하고 복기한다"는 서비스 본질은 증권사와 무관합니다.
  * 한국투자증권과 토스증권 역시 리테일 점유율 최상위권이므로, 초기 SOM(각성적 학습자 3,000명) 확보에는 무리가 없습니다. 서버리스 프레임워크 유지로 얻는 '운영비용 최소화 및 개발 속도'의 이득이 압도적으로 큽니다.

### 3) 자동 매매 복기 일지 (F3)
* **변경점:** 파이썬(Python) 기반 별도 데이터 분석 서버 구축을 폐기하고, **Vercel AI SDK + Google Gemini API**를 Next.js Server Actions 내에 직접 통합.
* **가치 훼손 여부: 오히려 가치 극대화 (UX 향상)**
  * 자체 모델이나 휴리스틱 로직을 설계하고 튜닝하는 유지보수 비용이 사라집니다.
  * Gemini API 특유의 대규모 컨텍스트 및 우수한 추론 능력을 활용해, 사용자의 매매 타점 데이터와 당일 매크로 뉴스를 입체적으로 엮어 '인사이트 넘치는' 일지를 즉각 생성할 수 있습니다. 3분 이내 스캔 가능한 고품질 UI 제공에 매우 유리합니다.

## 2. 단계별 구현 마일스톤 (Milestones)

### Phase 1: 기반 인프라 구축 및 공통 UI 설정 (Week 1)
* **작업 항목:**
  1. `Next.js 15+ (App Router)` 프로젝트 초기화.
  2. `Tailwind CSS` 및 `shadcn/ui` 적용하여 디자인 시스템(컴포넌트 라이브러리) 구축.
  3. `Supabase` 프로젝트 생성 및 Database Schema(`USER`, `ACCOUNT`, `TRADE` 등) 생성 및 RLS 정책 적용.
  4. Supabase Auth 연동을 통한 로그인/회원가입 기능 구현.
  5. Github 연동을 통한 Vercel 자동 배포 파이프라인(CI/CD) 구축.

### Phase 2: 증권사 REST API 연동 및 데이터 동기화 (Week 2)
* **작업 항목:**
  1. 한국투자증권(KIS Developers) 및 토스증권 API 연동 인터페이스 설계 (Server Actions).
  2. 증권사 OAuth 인증 및 인증키 저장 (DB 저장 시 AES-256 암호화 적용).
  3. Vercel Cron Jobs을 활용한 1~5분 주기 자동 동기화 트리거(`api/cron/sync`) 구현.
  4. 매매 데이터(`TRADE`)를 Supabase PostgreSQL에 적재하는 파이프라인 완성.

### Phase 3: 뇌동매매 감시 및 통계 대시보드 (Week 3)
* **작업 항목:**
  1. 사용자 손실폭/매매횟수 상한 설정 인터페이스(`ALERT_RULE`) 구현.
  2. 프론트엔드 폴링 + Supabase Realtime을 연계한 **실시간 손실 감시 및 팩트폭행 모달 알람** 구현.
  3. 알람 발생 시 `ALERT_LOG` 적재 및 매수 쿨타임 타이머 로직(클라이언트 단 버튼 비활성화) 처리.
  4. Supabase 데이터를 기반으로 승률, MDD 등을 계산하여 렌더링하는 `Recharts` 통계 대시보드(F5) 구축.

### Phase 4: AI 기반 자동 복기 일지 고도화 (Week 4)
* **작업 항목:**
  1. 장 마감(오후 3시 30분 이후) Vercel Cron 트리거로 일지 생성 로직 시작.
  2. `Vercel AI SDK` 초기화 및 `Google Gemini API` 연동.
  3. 당일 매매 내역(JSON)과 뉴스 데이터를 Gemini 모델의 프롬프트로 주입하여 핵심 인사이트 추출.
  4. `TRADE_REVIEW` 테이블에 반환된 구조화 데이터 저장 및 `Supabase Realtime`으로 클라이언트에 일지 생성 완료 알림 전송.
  5. 복기 일지 상세 조회 UI 렌더링.

## 3. 핵심 아키텍처 원칙 (C-TEC 준수)
1. **API 계층 통합:** 데이터베이스 조회 및 외부 API(증권사) 통신은 오직 Next.js **Server Actions 및 Route Handlers** 내부에서만 수행합니다. 클라이언트(브라우저)에서 DB로 직접 접근하거나 민감한 토큰을 다루지 않습니다.
2. **AI 연동 표준화:** LLM 교체 용이성을 위해 Vercel AI SDK의 `generateText` 또는 `generateObject` 등 표준 인터페이스만을 사용하여 Gemini 로직을 작성합니다.
3. **디자인 강제성:** 모든 프론트엔드 컴포넌트는 커스텀 CSS를 최대한 지양하고 `shadcn/ui`의 기본 패턴과 `Tailwind CSS`의 유틸리티 클래스를 엄격하게 준수합니다.
