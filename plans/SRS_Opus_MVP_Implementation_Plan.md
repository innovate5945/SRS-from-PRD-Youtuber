## 2. 핵심 사용자 경험 (가치 전달) 훼손 여부 검토

### 2.1 검토 원칙

MVP의 핵심 가치는 PRD 북극성 KPI인 **"WAU 중 주간 뇌동매매 0회 유저 비율 ≥ 40%"** 달성이다. 기술 스택 변경이 이 가치 전달 경로를 훼손하는지 기능별로 검토한다.

### 2.2 F1 — 뇌동매매 방지 강제 제어

| 항목 | 상세 |
|---|---|
| **변경 내용** | 상시 데몬 → 클라이언트 폴링 + 이메일 알람 발송 |
| **가치 전달 경로** | 손실폭 설정 → 장중 감시 → 임계치 도달 시 팩트폭행 알람 → 쿨타임 매수 차단 |
| **판정: ✅ 훼손 없음** | — |

**근거:**
- 사용자가 웹앱을 활성 상태로 두고 매매하는 시나리오에서, 클라이언트 측 폴링이 손실폭을 감지하고 즉각 모달 표시 및 이메일 알림을 발송한다. 기존 "3초 이내" 요건은 "감지 주기 내 즉각 발동"으로 조정되었으나, **뇌동매매 차단이라는 행동 경제학적 넛지(Nudge) 효과는 동일하게 작동**한다.
- 쿨타임(30분) 매수 버튼 비활성화는 서버 측 `ALERT_LOG` 기록 + 클라이언트 상태 관리로 100% 동일하게 구현 가능하다.
- 주간 뇌동매매 리포트는 Vercel Cron(주말 트리거)으로 완벽 대체된다.

### 2.3 F2 — 멀티 증권사 API 통합 연동

| 항목 | 상세 |
|---|---|
| **변경 내용** | 키움·NH 제외 → 한국투자·토스·LS (순수 REST API 지원 증권사) |
| **가치 전달 경로** | 온보딩 60초 내 연동 → 5분 주기 자동 동기화 → 통합 대시보드 합산 |
| **판정: ⚠️ 일부 리스크 있으나 수용 가능** | — |

**근거:**
- **키움증권은 Windows OCX 기반**이므로 Linux 서버리스(Vercel) 환경에서 물리적으로 호출 불가. 별도 Windows 중계 서버를 두면 C-TEC-001(단일 풀스택) 원칙에 위배된다.
- 한국투자증권(KIS Developers)은 2024년 기준 리테일 점유율 상위이며, 토스증권은 MZ세대(타겟 페르소나 연령대) 점유율 1위급이다.
- "자신의 매매를 통제하고 복기한다"는 서비스 본질은 증권사와 무관하며, CSV 수동 업로드 폴백(DEP-01)이 키움 사용자를 커버한다.
- **Kill Criteria:** Beta 중 미지원 증권사 요청 > 30% 시 어댑터 패턴(REQ-NF-032)으로 삼성·미래에셋 추가.

### 2.4 F3 — 자동 매매 복기 일지

| 항목 | 상세 |
|---|---|
| **변경 내용** | 자체 휴리스틱 매칭 로직 (MVP) + AI SDK 연동 준비 (Phase 2) |
| **가치 전달 경로** | 장 마감 → Cron 트리거 → 매매+뉴스 데이터 Gemini 분석 → 3분 내 스캔 가능 UI |
| **판정: ✅ 오히려 가치 향상** | — |

**근거:**
- MVP에서는 자체 규칙 기반 타점-뉴스 매칭을 우선 적용하며, Vercel AI SDK 연동은 인프라 구성만 셋업/테스트한다. 본격적인 고품질 인사이트 생성(Gemini API)은 Phase 2 확장에서 도입하여 PRD 목표인 3분(p50) 복기 달성을 극대화한다.

### 2.5 F5 — 통계 대시보드 + 매매횟수 브레이크

| 항목 | 상세 |
|---|---|
| **변경 내용** | 별도 Dashboard Aggregator 서비스 → Next.js Server Component + Supabase 쿼리 |
| **가치 전달 경로** | 대시보드 접속 → 승률·MDD·매매횟수 시각화 (p95 ≤ 2초) → 매매횟수 브레이크 |
| **판정: ✅ 훼손 없음** | — |

**근거:**
- Supabase PostgreSQL의 집계 쿼리 성능은 SOM 5,000명 규모에서 p95 ≤ 500ms를 충분히 달성한다.
- Next.js Server Components의 스트리밍 렌더링으로 대시보드 초기 로딩 체감 속도가 오히려 향상된다.
- 매매횟수 브레이크 알람은 F1과 동일하게 클라이언트 폴링과 이메일 알림, 인앱 모달을 활용하여 일관된 UX를 제공한다.

### 2.6 종합 판정

| 기능 | 판정 | 비즈니스 KPI 영향 |
|---|---|---|
| F1 뇌동매매 방지 | ✅ 훼손 없음 | 북극성 KPI 직접 기여 유지 |
| F2 증권사 연동 | ⚠️ 키움 제외 (폴백 존재) | D7 리텐션에 미미한 영향 |
| F3 자동 복기 | ✅ 가치 향상 | 복기 시간 3분 목표 초과 달성 가능 |
| F5 대시보드 | ✅ 훼손 없음 | 매매횟수 50% 감축 목표 유지 |

> **결론: MVP 핵심 가치(감정 제어 → 뇌동매매 0회 달성)는 기술 스택 전환 후에도 완전히 보존되며, F3(자동 복기)은 오히려 품질이 향상된다.**

## 4. 아키텍처 결정 기록 (ADR)

### ADR-001: 증권사 어댑터 패턴

- **결정:** `BrokerAdapter` 인터페이스를 정의하고, 각 증권사별 구현체(`KoreaInvestmentAdapter`, `TossAdapter`, `LsAdapter`)를 분리한다.
- **근거:** REQ-NF-032(신규 증권사 추가 시 기존 코드 변경 없이 확장) 충족.
- **구현 위치:** `lib/brokers/` 디렉토리.

### ADR-002: 이메일 중심 알람 시스템

- **결정:** 기존 Push 알림 및 SSE/웹소켓 방식 대신 이메일 발송(예: Resend 등)과 클라이언트 폴링을 활용한다.
- **근거:** Vercel Hobby(무료) 환경 및 MVP 단계에서 복잡한 실시간 연결 구성(Supabase Realtime 동시 접속 한도 등)을 배제하고, 가장 안정적이고 확실한 채널인 이메일을 우선 사용한다.
- **트레이드오프:** 인앱 실시간 Push보다 전달이 조금 지연될 수 있으나, 클라이언트 폴링을 통한 즉각적인 모달 노출로 보완한다.

### ADR-003: 자체 매칭 (MVP) 및 AI SDK 인프라 준비 (Phase 2)

- **결정:** MVP는 자체 규칙 기반 로직으로 복기 일지를 생성하고, 본격적인 LLM(Gemini) 분석은 Phase 2 확장을 위해 Vercel AI SDK 연동 인프라 셋업만 진행한다.
- **근거:** 초기 MVP의 런타임 불확실성을 최소화하고 무료 플랜(Hobby)의 비용을 통제하기 위해 LLM을 직접 호출하는 로직을 일차적으로 분리한다. 추후 준비된 SDK로 환경 변수만 변경하여 스위칭한다.
